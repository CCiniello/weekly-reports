01Broken pages (4xx / 5xx)~5,835 URLs · −14Error
Why it matters

Every broken URL is a dead end. Google only spends so much attention crawling our site, and thousands of dead ends burn that budget instead of landing on the pages we want ranked. It's the single biggest reason our score is 36.

What these actually are: each one is a whole URL — a complete web address — that returns an error instead of a page. (A "4xx" is always the entire address, never "part of a page" — you can't half-break a page this way.) Two kinds: 404 = "not found" (address doesn't exist) and 403 = "forbidden" (server refuses it). The ~5,835 figure counts broken sub-resources too; the core internal broken pages are 731 (+9), of which 420 are hard 404s (+5) and 1 is a 5xx. They sort into the three named causes in cards 02–04.

See real example URLs
  • https://www.revealbi.io/ja/embedded-analytics-software 403
  • https://www.revealbi.io/ko/license-agreements 403
  • https://wp-staging.revealbi.io/academy 404
  • https://www.revealbi.io/support@revealbi.io 403
  • https://help.revealbi.io/api/java/latest/.../RVAthenaDataSourceItem.html 404

Full list: Ahrefs → Site Audit (1241581) → Page Explorer → filter "4XX page."

Likely why: leftover URLs from the WordPress→Astro migration that were never redirected, plus the three causes below. Most of this number collapses once cards 02–04 are fixed. Note: a handful of real blog/author pages (e.g. /author/jasonb, /blog/boost-customer-retention…) are also returning 403 — likely the same server/firewall rule catching more than just the locale paths, so check breadth when fixing card 04.

The fix — step by step
  1. Fix the three root causes below first (staging copy, email bug, localized 403s) — they're the large majority.
  2. Re-crawl, then export whatever 4xx URLs remain.
  3. For each remaining one: if a live equivalent exists, 301 to it; if it's gone for good, return 410 so Google stops retrying.
  4. Remove internal links that still point at dead URLs.

03The support-email link bug is manufacturing broken URLssitewideRoot cause
Why it matters

A single one-line coding mistake is generating fake broken pages on every page of the site — inflating our error count for no reason. Tiny fix, disproportionately large cleanup.

What this actually is: the support email is written as a normal link (href="support@revealbi.io") instead of an email link (href="mailto:support@revealbi.io"). Missing the mailto:, the browser and Google read it as a page and try to visit revealbi.io/support@revealbi.io — which doesn't exist, so it errors. It's in the footer/contact area of nearly every page, so it multiplies fast — and it's now showing up on the staging copy too.

See real example URLs
  • https://www.revealbi.io/support@revealbi.io 403
  • https://www.revealbi.io/glossary/support@revealbi.io 403
  • https://wp-staging.revealbi.io/support@revealbi.io 404

Same email, rendered as a path on many pages → many broken URLs.

Likely why: a template/component where the email was pasted as a link without the mailto: prefix.

The fix — step by step
  1. Find where the support email is output (footer or contact component).
  2. Change href="support@revealbi.io" to href="mailto:support@revealbi.io" — or show it as plain text.
  3. Search the codebase for other bare email-as-link instances; fix the same way.
  4. Redeploy and re-crawl.

A canonical tag is a page telling Google "this is the official version of me." On 384 pages that pointer aims at a dead or moved URL — so we're telling Google the real version of these pages is broken, and Google may drop them from search. It ticked up again this week.

What this actually is: 380 pages have a canonical pointing at a broken (4xx) URL (+5), 3 point at a redirect (all new this week), and 1 points at a server error (5xx). Think of it as a page handing Google a forwarding address that leads nowhere.

6 Cheat sheetevery term above, in plain English

Health score
Ahrefs' rough 0–100 grade for the site's technical shape. Directional, not precise — a low score means lots of errors on the pages it crawled. Watch the trend.move it up over time
4xx vs 5xx (broken pages) — and "is it the whole page?"
A 4xx/5xx is always the whole URL (the whole address), never part of a page — you can't 404 half a page. 404 = not found. 403 = forbidden (server refuses access). 5xx = server failed to deliver it. All are dead ends that waste Google's crawl attention.fix: web dev
Canonical tag
A hidden tag telling Google "this is the official version of this page," used when similar pages exist. If it points at a broken or redirecting URL, you're telling Google the real version is dead/moved — so Google may drop the page.fix: web dev
Noindex / nofollow
Noindex = "do not put this in search results." Nofollow = "don't pass ranking credit through this page's links." Fine when intentional; harmful when it hits pages we want found.fix: web dev / CMS
XML sitemap
The clean list of URLs we hand Google saying "these are my real pages." It should contain only live, canonical, indexable pages — never broken or redirecting ones.fix: web dev
Redirect (3xx) & redirect chain
A redirect forwards one URL to another (used when a URL changes). Normal in small numbers. A chain is A→B→C — several hops — which is slow and leaks a little value each hop. Link straight to the final URL.fix: web dev
Crawl budget
The finite attention Google spends crawling our site. Every dead end, redirect, and junk URL eats into it — so broken pages literally cost our good pages visibility. The "why it matters" behind most of this report.the core idea
Non-production subdomain (staging / dev / old)
A test/pre-launch copy of the site (wp-staging., dev., old.). Should never be reachable by Google — if it is, it competes with the real site as duplicate content.fix: web dev / infra
mailto: link
The correct way to code an email as a clickable link (mailto:name@site.com). Without mailto:, the browser treats it as a page on your site — the bug in card 03.fix: web dev
hreflang
A tag that tells Google which language/region version of a page to show which visitors (e.g. the Japanese version to searchers in Japan). If it points at a broken, redirecting, or non-canonical URL, Google gets confused signals and may show the wrong version — or none. The issue in card 09.fix: web dev
